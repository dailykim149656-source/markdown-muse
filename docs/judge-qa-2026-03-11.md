# Docsy Judge Q&A Notes

## What does the product do?

Docsy is an AI document workflow agent for technical documentation. It helps users maintain consistency across related docs by proposing reviewable patches instead of directly editing files.

## Where is Gemini actually used?

Gemini is used in the AI service through the Google GenAI SDK.

Current uses:

- structured summaries
- section generation
- TOC suggestion
- structured action proposal
- multimodal request path with image payload plus document context

## What makes this different from a normal AI editor?

Most AI editors produce text. Docsy produces workflow actions and reviewable patch proposals. The user stays in control and reviews changes before applying them.

## What is the multimodal part?

The editor sends document context plus an image payload representing the current editor state. That image is included alongside the prompt in the Gemini request.

## What structured output do you use?

Gemini returns strict JSON. In the current demo, one action is wired end-to-end:

- `open_patch_review`

That JSON output is mapped to a real editor behavior.

## Why is review-first important?

Technical documentation is operational and often high-impact. Silent AI rewriting is risky. A review-first workflow makes the AI useful without removing user control.

## What happens in the demo?

1. A procedure is changed in one document.
2. Docsy recommends related documents and highlights conflicts.
3. The app sends multimodal context to Gemini.
4. Gemini returns action JSON and a patch proposal.
5. Patch review opens.
6. The user accepts or rejects the change.

## Is this just prompt engineering?

No. The project includes:

- editor workflows
- document structure handling
- knowledge and relationship signals
- strict response schemas
- UI behavior driven by model output
- review and patch application flow

## What parts are already implemented?

- related-document recommendation
- conflict highlighting
- TOC suggestion preview
- before/after patch review cards
- Cloud Run-ready AI service path
- submission and demo documentation

## What is still limited?

- the current multimodal image is a generated snapshot rather than full DOM capture
- one structured action is wired today
- deployment still needs real project-specific GCP values

## Why should this score well?

- it uses Gemini in a practical product workflow
- it demonstrates multimodal input and structured output
- it ties model output to a real UI action
- it solves a clear documentation maintenance problem
